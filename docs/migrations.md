# Database Migrations

This document describes the formal database migration system for the scoopdope backend, including versioned migrations, rollback support, and programmatic execution.

## Overview

The application uses TypeORM migrations to manage database schema changes. Key design decisions:

- **Versioned migrations**: Each migration is a timestamped file with `up` and `down` methods
- **Rollback support**: Every migration includes a `down()` for safe reversal
- **Dedicated migrations table**: `schema_migrations` tracks applied migrations
- **Programmatic runner**: A `MigrationRunnerService` enables migration execution via API or CLI
- **Transaction mode**: All migrations run in a transaction (`migrationsTransactionMode: 'all'`)
- **`synchronize: false`**: Disabled in all environments to prevent accidental schema changes

## API Endpoints (Admin Only)

| Method | Path | Description |
|---|---|---|
| `GET` | `/v1/database/migrations` | List all migrations with status |
| `POST` | `/v1/database/migrations/run` | Execute all pending migrations |
| `POST` | `/v1/database/migrations/revert` | Revert the last migration |

All endpoints require `JwtAuthGuard` + `RolesGuard` with `@Roles('admin')`.

## CLI Scripts

```bash
# Run all pending migrations
npm run migration:run

# Revert the last migration
npm run migration:revert

# Show migration status (executed + pending)
npm run migration:show

# Create an empty migration file
npm run migration:create -- NAME=AddNewFeature

# Generate a migration from entity changes
npm run migration:generate -- NAME=AutoGeneratedMigration
```

The CLI runner (`apps/backend/src/database/migration-runner.ts`) initializes the `AppDataSource` directly and executes the requested command. It is independent of the NestJS application, making it suitable for Docker entrypoints and CI/CD pipelines.

## Programmatic Execution (main.ts)

The application entry point supports migration commands as CLI arguments:

```bash
# Run via the compiled app
node dist/main migration:run
node dist/main migration:revert
```

This is used in production deployment scripts where the compiled NestJS application is already available.

## Migration File Structure

Migration files are located in `src/migrations/` and follow the naming convention:

```
src/migrations/<timestamp>-<Description>.ts
```

Each migration exports a class implementing `MigrationInterface`:

```typescript
import { MigrationInterface, QueryRunner, TableColumn } from 'typeorm';

export class AddCourseCategory1700000000000 implements MigrationInterface {
  name = 'AddCourseCategory1700000000000';

  public async up(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.addColumn(
      'courses',
      new TableColumn({
        name: 'category',
        type: 'varchar',
        length: '50',
        isNullable: true,
      }),
    );
  }

  public async down(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.dropColumn('courses', 'category');
  }
}
```

## Workflows

### Development Workflow

1. **Make entity changes**: Modify your TypeORM entities
2. **Generate migration**: `npm run migration:generate -- NAME=DescriptiveName`
3. **Review migration**: Check the generated file for correctness
4. **Apply locally**: `npm run migration:run`
5. **Commit**: Commit both entity changes and the migration file

### Production Deployment Workflow

1. **Backup database**: Always create a backup before running migrations
2. **Deploy code**: Deploy the new Docker image including migration files
3. **Run migrations**: `node dist/main migration:run`
4. **Verify**: Check the migration status via `POST /v1/database/migrations` API
5. **Monitor**: Watch for errors in application logs

### Rollback Workflow

1. **Identify issue**: Determine which migration caused the problem
2. **Revert**: `node dist/main migration:revert` (can be run multiple times)
3. **Fix**: Create a new migration with the correction
4. **Re-apply**: Run migrations again

If a rollback is needed in production:

```bash
# Via Docker
docker run --rm --env-file .env.production \
  ghcr.io/scoopdope/backend:<previous-tag> \
  node dist/main migration:revert

# Via npm
npm run migration:revert
```

## Configuration

The migration system is configured in `apps/backend/src/data-source.ts`:

| Setting | Value | Description |
|---|---|---|
| `migrationsTableName` | `schema_migrations` | Dedicated migrations table |
| `migrationsTransactionMode` | `all` | Each migration runs in a transaction |
| `synchronize` | `false` | Never auto-sync schema |
| `logging` | `['error', 'schema']` | Log errors and schema changes |

Environment variables:

| Variable | Default | Description |
|---|---|---|
| `DATABASE_HOST` | `localhost` | Database host |
| `DATABASE_PORT` | `5432` | Database port |
| `DATABASE_USER` | required | Database user |
| `DATABASE_PASSWORD` | required | Database password |
| `DATABASE_NAME` | `scoopdope` | Database name |
| `DB_POOL_SIZE` | `10` | Connection pool size |
| `DB_IDLE_TIMEOUT` | `10000` | Idle connection timeout |
| `DB_LOGGING` | `false` | Enable query logging |
| `DB_SSL_REJECT_UNAUTHORIZED` | `true` | SSL certificate validation |

## Migration Runner Service

The `MigrationRunnerService` (`apps/backend/src/database/migration-runner.service.ts`) can be injected into other services:

```typescript
@Injectable()
export class MyService {
  constructor(private readonly migrationRunner: MigrationRunnerService) {}

  async deploy() {
    await this.migrationRunner.run();
  }
}
```

## Best Practices

1. **One concern per migration**: Each migration should handle a single logical change
2. **Always write down methods**: Every migration must be reversible
3. **Test on staging**: Always test migrations with a production-size dataset
4. **Backup before production**: Never run migrations without a verified backup
5. **Review generated code**: Always review auto-generated migrations before committing
6. **Keep migrations small**: Large migrations are harder to rollback and debug
7. **Use transactions**: All migrations are already wrapped in transactions

## Troubleshooting

### Migration fails with "relation already exists"

The database schema may be out of sync with migrations:

1. Check if the table/column already exists manually
2. Create a new migration to reconcile the schema
3. Use `DROP IF EXISTS` / `IF NOT EXISTS` guards for idempotency

### Migration fails in production

1. Check database logs for specific error messages
2. Verify database credentials and permissions
3. Ensure database is accessible from the application server
4. Rollback via `node dist/main migration:revert`
5. If the migration was partially applied, manual cleanup may be needed

### Generated migration is empty

TypeORM detected no changes between entities and database:

1. Verify database connection is working
2. Ensure entities are properly loaded
3. Confirm you've made actual changes to entities
4. Check that TypeORM can detect the changes (e.g., proper decorators)
